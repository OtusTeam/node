# Конфигурация Jest с SWC

Этот проект использует **SWC (Speedy Web Compiler)** вместо `ts-jest` для более быстрой компиляции TypeScript в тестах.

## Основные преимущества SWC

- ⚡ **Быстрее в 20+ раз** чем ts-jest
- 🚀 **Меньше потребление памяти**
- 🔧 **Поддержка декораторов и метаданных** (важно для NestJS)
- 📦 **Меньший размер бандла**

## Структура конфигурации

### `jest.config.js` - Основной конфиг для unit тестов

```javascript
transform: {
  '^.+\\.(t|j)sx?$': ['@swc/jest', { ... }]
}
```

**Ключевые настройки:**

1. **Покрытие кода (Code Coverage)**
   - Порог покрытия: 70% для branches, functions, lines, statements
   - Исключения: DTO, схемы, модули, интерфейсы
   - Форматы: text, html, lcov, json

2. **Параллельное выполнение**
   - `maxWorkers: '50%'` - использует половину доступных CPU

3. **Очистка моков**
   - `clearMocks: true` - очищает моки между тестами
   - `restoreMocks: true` - восстанавливает оригинальные реализации

4. **Таймауты**
   - `testTimeout: 10000` - 10 секунд для unit тестов

### `test/jest-e2e.json` - Конфиг для E2E тестов

- Использует SWC трансформер
- `maxWorkers: 1` - последовательное выполнение (важно для БД)
- `testTimeout: 30000` - 30 секунд для E2E тестов
- `verbose: true` - детальный вывод

## Доступные команды

```bash
# Базовые команды
npm test              # Запуск всех unit тестов
npm run test:watch    # Watch режим (автоматический перезапуск)
npm run test:cov      # С покрытием кода
npm run test:e2e      # E2E тесты

# Дополнительные команды
npm run test:verbose  # Детальный вывод всех тестов
npm run test:only     # Запуск тестов по паттерну имени
npm run test:file     # Запуск тестов для измененных файлов
```

## Примеры использования

### Запуск конкретного теста

```bash
npm test -- users.service.spec.ts
```

### Запуск тестов по паттерну

```bash

```


## Настройки покрытия кода

### Пороги покрытия

В `jest.config.js` установлены пороги:
- **branches**: 70% - покрытие всех веток условий
- **functions**: 70% - покрытие всех функций
- **lines**: 70% - покрытие всех строк
- **statements**: 70% - покрытие всех выражений

Если покрытие ниже порога, тесты упадут с ошибкой.

### Исключения из покрытия

Следующие типы файлов исключены:
- `*.spec.ts` - тестовые файлы
- `*.dto.ts` - Data Transfer Objects
- `*.schema.ts` - схемы Mongoose
- `*.module.ts` - модули NestJS
- `*.interface.ts` - интерфейсы TypeScript
- `main.ts` - точка входа приложения

## Расширенные возможности

### 1. Глобальные настройки (`test/setup.ts`)

Файл выполняется перед каждым тестом. Можно использовать для:
- Настройки глобальных моков
- Конфигурации окружения
- Инициализации тестовых данных

### 2. Watch режим

```bash
npm run test:watch
```

Режимы:
- `a` - запустить все тесты
- `f` - запустить только проваленные тесты
- `q` - выйти из watch режима
- `p` - фильтр по имени файла
- `t` - фильтр по имени теста

### 3. Отладка тестов

```bash
npm run test:debug
```

Запускает тесты в режиме отладки. Можно подключиться через Chrome DevTools:
1. Откройте `chrome://inspect`
2. Нажмите "Open dedicated DevTools for Node"

### 4. Параллельное выполнение

Jest автоматически запускает тесты параллельно. Для E2E тестов используется `maxWorkers: 1` для избежания конфликтов с БД.

## Сравнение производительности

| Трансформер | Время компиляции | Память |
|------------|------------------|--------|
| ts-jest    | ~5-10 сек        | ~200MB |
| @swc/jest  | ~0.5-1 сек       | ~50MB  |

## Полезные советы

1. **Используйте `test.only()`** для отладки одного теста
2. **Используйте `test.skip()`** для временного пропуска тестов
3. **Группируйте связанные тесты** с `describe()`
4. **Используйте `beforeEach`/`afterEach`** для настройки/очистки
5. **Проверяйте покрытие регулярно** - `npm run test:cov`

## Troubleshooting

### Проблема: Тесты не находят модули

Решение: Проверьте `moduleNameMapper` в `jest.config.js`

### Проблема: Декораторы не работают

Решение: Убедитесь, что в SWC конфиге включены:
```javascript
transform: {
  legacyDecorator: true,
  decoratorMetadata: true,
}
```

### Проблема: Медленные тесты

Решение:
- Используйте моки вместо реальных зависимостей
- Уменьшите `maxWorkers` для E2E тестов
- Используйте `test.only()` для отладки
